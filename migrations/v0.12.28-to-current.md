Apply BC entries in order first, then read AD entries to learn the new API surface.
See `migrations/README.md` for format details and the full LLM workflow.

## Breaking Changes

### BC-01 ModuleLogic type parameter removed

**Type:** Breaking

**Grep:** `: ModuleLogic<`
**File glob:** `**/*.kt`

**Before:**
```kotlin
class MyLogic(dispatch: Dispatch) : ModuleLogic<MyAction>(dispatch) { ... }

val createLogic: (Dispatch) -> ModuleLogic<MyAction> = { MyLogic(it) }
```

**After:**
```kotlin
class MyLogic(dispatch: Dispatch) : ModuleLogic(dispatch) { ... }

val createLogic: (Dispatch) -> ModuleLogic = { MyLogic(it) }
```

**Notes:** Also remove the type argument from any `-> ModuleLogic<X>` return type
annotations. `ModuleLogic` is no longer generic.

### BC-02 ModuleLogic lambda factory removed

**Type:** Breaking

**Grep:** `ModuleLogic {`
**File glob:** `**/*.kt`

**Before:**

```kotlin
override val createLogic = { dispatch: Dispatch ->
    ModuleLogic { action ->
```

**After:**

**Notes:** The anonymous `ModuleLogic { action -> }` lambda constructor no longer

### BC-03 ModuleLogic.invoke() override removed

**Type:** Breaking

**Grep:** `override suspend fun invoke(`
**File glob:** `**/*.kt`

**Before:**

**After:**

**Notes:** `ModuleLogic` no longer acts as a function receiver. Remove all

### BC-04 onStoreReset() renamed to beforeReset()

**Type:** Breaking

**Grep:** `onStoreReset()`
**File glob:** `**/*.kt`

**Before:**
```kotlin
override suspend fun onStoreReset() {
    // cleanup called after reinitialization
}
```

**After:**
```kotlin
override suspend fun beforeReset() {
    // cleanup called BEFORE reinitialization — cancel jobs, release resources
}
```

**Notes:** Semantics changed: `beforeReset()` is called *before* modules are
re-initialized, giving logic a chance to cancel coroutines and release resources.
Any code that assumed modules were already reset must be moved earlier or
restructured. See AD-10 for the full lifecycle documentation.

## Additions

### AD-01 createNavigationModule { } DSL

**Type:** Addition

**Grep:** `createNavigationModule`
**File glob:** `**/*.kt`

**Example:**
```kotlin
val navModule = createNavigationModule {
    // Required: define the root graph
    rootGraph {
        // entry() sets the start destination for this graph and registers the screen
        entry(SplashScreen)
        // screens() registers additional screens reachable within this graph
        screens(SplashScreen)

        // Named sub-graphs group related screens
        graph("main") {
            entry(HomeScreen)
            screens(HomeScreen, DetailScreen, ProfileScreen)
        }

        graph("auth") {
            entry(LoginScreen)
            screens(LoginScreen, SignupScreen)
        }
    }

    // Optional: modal shown during async guard / entry-route evaluation
    loadingModal(AppLoadingModal)

    // Optional: deep link aliases map external URL patterns to internal routes
    deepLinkAliases {
        alias(
            pattern = "myapp://home",
            targetRoute = "main/home"
        )
        alias(
            pattern = "myapp://product/{id}",
            targetRoute = "main/detail"
        )
    }
}
```

**Notes:** Replaces the direct `NavigationModule(...)` constructor (see BC-18).
All graph structure, loading modals, and deep link configuration live inside
this DSL block. The returned module is used in `createStore { module(navModule) }`.
For the full `entry()` DSL (including dynamic screen selection), see AD-14.
For the full `deepLinkAliases` DSL, see AD-12.

### AD-02 intercept { } guard DSL + GuardResult

**Type:** Addition

**Grep:** `intercept(`
**File glob:** `**/*.kt`

**Example:**
```kotlin
createNavigationModule {
    rootGraph {
        // Wrap any graph block with intercept() to add an access guard
        intercept(
            guard = { store ->
                val auth = store.selectState<AuthState>()
                when {
                    auth.isLoggedIn -> GuardResult.Allow
                    auth.isPending -> GuardResult.Reject
                    else -> GuardResult.PendAndRedirectTo(LoginScreen.route)
                }
            }
        ) {
            graph("protected") {
                screen(DashboardScreen)
                screen(SettingsScreen)
            }
        }

        // Unguarded screens sit outside the intercept block
        screen(LoginScreen)
    }
}
```

**GuardResult options:**
```kotlin
GuardResult.Allow                          // navigation proceeds
GuardResult.Reject                         // navigation blocked, no redirect
GuardResult.RedirectTo(route: String)      // navigate elsewhere instead
GuardResult.PendAndRedirectTo(route: String) // store intent, redirect for now;
                                             // resume later with resumePendingNavigation()
```

**Notes:** Guards receive a `StoreAccessor` so they can read any module state.
`PendAndRedirectTo` is the login-wall pattern — redirect the user to login, then
call `store.resumePendingNavigation()` after authentication succeeds to replay
the original navigation intent (see AD-08).

`intercept` blocks may be nested to form guard chains — see AD-13.

### AD-03 NavigationOutcome sealed class

**Type:** Addition

**Grep:** `NavigationOutcome`
**File glob:** `**/*.kt`

**Example:**
```kotlin
val outcome: NavigationOutcome = navLogic.navigate {
    navigateTo(DashboardScreen)
}

when (outcome) {
    is NavigationOutcome.Success ->
        // Navigation applied; outcome.entry is the new current entry
        log("navigated to ${outcome.entry.path}")

    is NavigationOutcome.Dropped ->
        // No-op: target was already the current screen
        Unit

    is NavigationOutcome.Rejected ->
        // Guard blocked the navigation; no redirect
        showError("Access denied")

    is NavigationOutcome.Redirected ->
        // Guard redirected elsewhere; outcome.to is the redirect target route
        log("redirected to ${outcome.to}")
}
```

**Notes:** The return value of `NavigationLogic.navigate()` (see BC-23). The
`StoreAccessor.navigation { }` extension discards the outcome for fire-and-forget
use cases.

### AD-10 ModuleLogic.beforeReset() lifecycle hook

**Type:** Replaces-deprecated

**Grep:** `beforeReset()`
**File glob:** `**/*.kt`

**Replaces:** `ModuleLogic.onStoreReset()` (renamed, see BC-04)

**Example:**
```kotlin
class MyLogic(dispatch: Dispatch) : ModuleLogic(dispatch) {
    private val job = SupervisorJob()
    private val scope = CoroutineScope(Dispatchers.Default + job)

    init {
        scope.launch { pollForUpdates() }
    }

    override suspend fun beforeReset() {
        // Cancel all coroutines before the store is re-initialized.
        // Called before initialState is restored and a new logic instance is created.
        job.cancel()
    }

    private suspend fun pollForUpdates() { ... }
}
```

**Notes:** `beforeReset()` is called before module state is restored to
`initialState` and before a new logic instance is constructed. This is the correct
place to cancel coroutines, close connections, and release resources. After
`beforeReset()` returns, the logic instance is discarded.

### AD-11 loadingModal — automatic overlay during async guard and entry evaluation

**Type:** Addition

**Grep:** `loadingModal`
**File glob:** `**/*.kt`

**Example:**
```kotlin
// Implement the LoadingModal interface — not Modal directly.
// LoadingModal already defaults dismissable=false, tapOutsideToDismiss=false,
// shouldDimBackground=false, and renderLayer=RenderLayer.SYSTEM.
object AppLoadingModal : LoadingModal {
    override val route = "app-loading"
    override val enterTransition = NavTransition.Fade
    override val exitTransition = NavTransition.FadeOut

    @Composable
    override fun Content(params: Params) {
        Box(Modifier.fillMaxSize(), contentAlignment = Alignment.Center) {
            CircularProgressIndicator()
        }
    }
}

// Register it in the navigation module
val navModule = createNavigationModule {
    rootGraph { ... }
    loadingModal(AppLoadingModal)
}
```

**When it appears:** Shown automatically when an `intercept` guard or a dynamic
`entry(route = { ... })` lambda takes longer than its configured `loadingThreshold`
to return (default 200 ms). Removed atomically once navigation commits.

**Render layer:** `LoadingModal` sits at `RenderLayer.SYSTEM` with `elevation = 0f`,
placing it *below* any user-triggered system modals (which use `elevation = 1000f`)
but above all `GLOBAL_OVERLAY` modals. See AD-16 for the full layer hierarchy.

**Notes:** Only one loading modal can be registered per navigation module. It does
not need to be registered in `screens()` or `modals()` — the navigation module
registers it automatically. Do not implement `Modal` directly for a loading overlay;
use `LoadingModal` to get the correct layer, elevation, and dismissal defaults.

### AD-15 notFoundScreen and crashScreen — root fallback screens

**Type:** Addition

**Grep:** `notFoundScreen\|crashScreen`
**File glob:** `**/*.kt`

**Example:**
```kotlin
val navModule = createNavigationModule {
    rootGraph { ... }

    // Shown when navigating to a route that doesn't exist in the graph
    notFoundScreen(NotFoundScreen)

    // Shown when a ModuleLogic method throws an unhandled exception
    crashScreen(
        screen = CrashScreen,
        onCrash = { exception, action ->
            // Optional: report the crash before navigation
            FirebaseCrashlytics.getInstance().recordException(exception)
            // Return one of two values:
            CrashRecovery.NAVIGATE_TO_CRASH_SCREEN  // show crashScreen, do NOT rethrow
            // CrashRecovery.RETHROW                // let the exception propagate normally
        }
    )
}
```

**CrashScreen params** — the crash screen receives these params automatically:
```kotlin
object CrashScreen : Screen {
    override val route = "crash"
    override val enterTransition = NavTransition.None
    override val exitTransition = NavTransition.None

    @Composable
    override fun Content(params: Params) {
        val exceptionType    = params["exceptionType"] as? String
        val exceptionMessage = params["exceptionMessage"] as? String
        val actionType       = params["actionType"] as? String
        // display error UI, offer retry / go home
    }
}
```

**Notes:** Both screens are optional. If `notFoundScreen` is not set, navigating to
an unknown route throws `RouteNotFoundException`. If `crashScreen` is not set,
unhandled exceptions in logic propagate normally. Neither screen needs to be
registered in `screens()` — the navigation module registers them automatically.

### AD-16 RenderLayer — content, global overlay, and system modals

**Type:** Addition

**Grep:** `RenderLayer\|renderLayer`
**File glob:** `**/*.kt`

**Layer hierarchy (bottom to top):**

| Layer | Constant | Default for | Renders above |
|---|---|---|---|
| Normal screens | `RenderLayer.CONTENT` | `Screen` | — |
| User-triggered modals | `RenderLayer.GLOBAL_OVERLAY` | `Modal` | all CONTENT |
| System overlays | `RenderLayer.SYSTEM` | `LoadingModal` | everything |

Within `GLOBAL_OVERLAY`, `elevation` controls stacking order (default `1000f`).
Within `SYSTEM`, `LoadingModal` uses `elevation = 0f` and user system modals use
`elevation = 1000f`, so user alerts always appear above the loading overlay.

**Regular modal (GLOBAL_OVERLAY)** — user-visible, dismissable by default:
```kotlin
object ConfirmDeleteModal : Modal {
    override val route = "confirm-delete"
    override val enterTransition = NavTransition.SlideUp
    override val exitTransition = NavTransition.SlideDown
    // renderLayer defaults to RenderLayer.GLOBAL_OVERLAY
    // dismissable defaults to true
    // tapOutsideToDismiss defaults to true

    @Composable
    override fun Content(params: Params) { ... }
}
```

**System modal (SYSTEM layer)** — rendered above everything, including other modals.
Use for force-update screens, critical error alerts, or consent gates that must
never be obscured:
```kotlin
object ForceUpdateModal : Modal {
    override val route = "force-update"
    override val enterTransition = NavTransition.Fade
    override val exitTransition = NavTransition.None
    override val renderLayer get() = RenderLayer.SYSTEM
    override val dismissable get() = false
    override val tapOutsideToDismiss get() = false
    override val shouldDimBackground get() = true

    @Composable
    override fun Content(params: Params) {
        // Full-screen force-update UI — cannot be dismissed
    }
}
```

Navigate to a system modal the same way as any other screen:
```kotlin
store.navigation { navigateTo(ForceUpdateModal) }
```

Pair with an outer `intercept` guard to show it automatically on any navigation
attempt when the condition is active:
```kotlin
intercept(
    guard = { store ->
        if (store.selectState<AppState>().value.forceUpdateRequired)
            GuardResult.RedirectTo(ForceUpdateModal)
        else
            GuardResult.Allow
    }
) {
    // all protected graphs
}
```

**Notes:** `LoadingModal` is a specialised `SYSTEM` modal managed automatically by
the navigation system — use it only for the async evaluation overlay (see AD-11).
For application-level system overlays (force update, critical errors, consent),
implement `Modal` directly and override `renderLayer` to `RenderLayer.SYSTEM`.

### AD-13 Nested intercept { } blocks chain guards (outer-first)

**Type:** Addition

**Grep:** `intercept(`
**File glob:** `**/*.kt`

**Example — two levels (startup gate → auth gate):**
```kotlin
createNavigationModule {
    rootGraph {
        entry(startScreen)
        screens(startScreen, loginScreen)

        intercept(
            guard = { store ->
                // Outer: system-level gate evaluated first
                if (store.selectState<AppState>().value.startupReady) GuardResult.Allow
                else GuardResult.Reject
            }
        ) {
            intercept(
                guard = { store ->
                    // Inner: only reached when outer returns Allow
                    if (store.selectState<AuthState>().value.isAuthenticated) GuardResult.Allow
                    else GuardResult.RedirectTo(loginScreen)
                }
            ) {
                graph("workspace") {
                    entry(homeScreen)
                    screens(homeScreen)
                }
            }
        }
    }
}
```

**Example — three levels + side-by-side independent chains:**
```kotlin
intercept(guard = { /* startup check */ }) {
    intercept(guard = { /* auth check */ }) {

        // workspace chain: startup → auth
        graph("workspace") {
            entry(homeScreen)
            screens(homeScreen)
        }

        // premium chain: startup → auth → premium (independent from workspace)
        intercept(guard = { /* premium check */ }) {
            graph("premium") {
                entry(premiumHome)
                screens(premiumHome)
            }
        }
    }
}
```

**Chaining rules:**
- Guards run in declaration order — outermost first.
- The first non-`Allow` result stops evaluation; inner guards are never called.
- Side-by-side `intercept` blocks at the same level are fully independent — each
  wrapped graph accumulates only the guards that apply to it.
- Nesting depth is unlimited.

**Notes:** Before this addition, nesting two `intercept {}` blocks caused the outer
guard to silently overwrite the inner one — only the outermost guard ran. Now the
full chain is preserved. Existing single-level intercepts are unaffected.

### AD-14 entry() DSL — static and dynamic graph start destinations

**Type:** Addition

**Grep:** `entry(`
**File glob:** `**/*.kt`

**Static form** — navigates directly to a fixed screen when the graph is entered:
```kotlin
graph("workspace") {
    entry(HomeScreen)
    screens(HomeScreen, ProfileScreen, SettingsScreen)
}
```

**Dynamic form** — the lambda receives a `StoreAccessor` and must return a
`NavigationNode`. Three return types are available:

| Return type | How | Example |
|---|---|---|
| Typed screen/modal object | Return the object directly | `HomeScreen` |
| Graph by route string | Wrap in `NavigationPath("route")` | `NavigationPath("home")` |
| Screen by route string | Wrap in `NavigationPath("route")` | `NavigationPath("home/dashboard")` |

**Returning a typed Navigatable directly:**
```kotlin
graph("content") {
    entry(
        route = { store ->
            val state = store.selectState<ContentState>().value
            when {
                state.hasReleases -> ReleasesScreen   // typed Screen object
                state.hasArtists  -> ArtistsScreen
                else              -> NoContentScreen
            }
        }
    )
    screens(ReleasesScreen, ArtistsScreen, NoContentScreen)
}
```

**Returning a graph reference via `NavigationPath`** — the logic navigates to that
route and then evaluates *that graph's own entry* recursively:
```kotlin
rootGraph {
    entry(
        route = { store ->
            val auth = store.selectState<AuthState>().value
            if (auth.isAuthenticated) NavigationPath("home")  // → resolves home's entry
            else LoginScreen                                   // → goes directly to LoginScreen
        }
    )
    screens(LoginScreen)

    graph("home") {
        startGraph("news")   // home's own entry — resolved after rootGraph's entry redirects here
        graph("news") { entry(NewsScreen); screens(NewsScreen) }
        graph("workspace") { entry(WorkspaceScreen); screens(WorkspaceScreen) }
    }
}
```

**Returning a route string via `NavigationPath`** — use when you don't have a typed
object available (e.g. the screen is in a different module):
```kotlin
entry(
    route = { store ->
        val ready = store.selectState<AppState>().value.onboardingComplete
        if (ready) NavigationPath("home/dashboard") else NavigationPath("onboarding")
    }
)
```

**loadingThreshold** — if the lambda takes longer than this to return, the global
loading modal is shown as an overlay (default 200 ms):
```kotlin
entry(
    loadingThreshold = 500.milliseconds,
    route = { store ->
        // suspend call — loading modal appears after 500ms if still running
        val session = store.selectLogic<AuthLogic>().checkSession()
        if (session.isValid) NavigationPath("home") else LoginScreen
    }
)
```

**Where it triggers:** `entry()` is evaluated when navigating *to the graph itself*
(e.g. `navigateTo("content")`), not when navigating directly to a screen inside
the graph (e.g. `navigateTo("content/releases")`). Direct screen navigation bypasses
the entry route and goes straight to the specified screen.

**Notes:** The static `entry(Screen)` form registers the screen as a navigatable
automatically. In the dynamic form, all screens that could be returned must still be
registered via `screens()` — the lambda return is not analysed statically.
`NavigationPath` is the bridge between the typed world (Screen/Modal objects) and
the route-string world (graph IDs, cross-module routes).

### AD-12 Built-in deep link alias system

**Type:** Addition

**Grep:** `deepLinkAliases`
**File glob:** `**/*.kt`

**Example:**
```kotlin
val navModule = createNavigationModule {
    rootGraph {
        entry(HomeScreen)
        screens(HomeScreen, DetailScreen, ProfileScreen)
    }

    deepLinkAliases {
        // Simple alias: pattern maps directly to a canonical internal route
        alias(
            pattern = "myapp://home",
            targetRoute = "home"
        )

        // Parameterised: {param} segments are extracted from the URL and forwarded
        alias(
            pattern = "myapp://product/{id}",
            targetRoute = "detail"
        )

        // With paramsMapping: transform or rename params before navigation
        alias(
            pattern = "legacy/invite/{code}",
            targetRoute = "workspace/invite/{token}"
        ) { params ->
            Params.of("token" to (params["code"] as? String ?: ""))
        }
    }
}

// Call this whenever the app receives a deep link URL
storeAccessor.navigateDeepLink("myapp://product/42")
// DetailScreen receives params = mapOf("id" to "42")
```

**Notes:** If the target project contains any of the following, this system is
likely a direct replacement:
- Manual URI parsing (`Uri.parse`, `intent.data`, URL splitting/regex)
- Hand-written URL-to-route mapping tables or `when`/`if` chains that inspect
  incoming URLs and call navigation functions
- Custom `DeepLinkHandler`, `DeepLinkRouter`, or similarly named classes
- `onNewIntent` / `handleIntent` methods that extract path segments and dispatch
  navigation actions directly

The built-in system handles URL parsing, parameter extraction, guard evaluation,
and the loading modal automatically. Guards defined with `intercept { }` (see AD-02)
apply to deep link navigation the same way they apply to in-app navigation. If a
guard returns `PendAndRedirectTo`, the deep link intent is stored and can be resumed
with `resumePendingNavigation()` after the precondition is satisfied (see AD-08).

### AD-21 BackstackLifecycle and onLifecycleCreated — per-screen lifecycle hooks

**Type:** Addition

**Grep:** `onLifecycleCreated\|BackstackLifecycle\|invokeOnRemoval`
**File glob:** `**/*.kt`

**When it fires:** `onLifecycleCreated` is called exactly once when a screen is added
to the backstack. After `Store.reset()`, it is called again for each entry that
remains in the backstack with a fresh lifecycle instance.

**Example — dispatch initial load, observe visibility, register cleanup:**
```kotlin
object ProjectScreen : Screen {
    override val route = "project"
    override val enterTransition = NavTransition.SlideInRight
    override val exitTransition = NavTransition.SlideOutLeft

    override suspend fun onLifecycleCreated(lifecycle: BackstackLifecycle) {
        // Dispatch initial action if this screen is the active one
        if (lifecycle.visibility.value) {
            lifecycle.dispatch(ProjectAction.Load(lifecycle.params["id"] as String))
        }

        // Launch a coroutine scoped to this entry's lifetime
        // Automatically cancelled when the entry leaves the backstack
        lifecycle.launch {
            lifecycle.selectState<ProjectState>().collect { state ->
                if (lifecycle.visibility.value) {
                    lifecycle.dispatch(ProjectAction.Update(state))
                }
            }
        }

        // Register cleanup invoked when the entry is removed from the backstack
        lifecycle.invokeOnRemoval { reason ->
            // `this` is StoreAccessor; reason is NAVIGATION or RESET
            launch {
                val logic = selectLogic<ProjectLogic>()
                logic.cleanup()
            }
        }
    }

    @Composable
    override fun Content(params: Params) { ... }
}
```

**BackstackLifecycle API:**

| Member | Type | Description |
|---|---|---|
| `entry` | `NavigationEntry` | The navigation entry for this screen |
| `params` | `Params` | Shorthand for `entry.params` |
| `route` | `String` | Shorthand for `entry.route` |
| `visibility` | `StateFlow<Boolean>` | `true` when this entry is the current screen |
| `dispatch(action)` | `Unit` | Dispatch an action to the store |
| `selectState<S>()` | `StateFlow<S>` | Observe any module state |
| `selectLogic<L>()` | `L` | Access any logic instance |
| `launch { }` | `Job` | Launch a coroutine scoped to this entry |
| `invokeOnRemoval { }` | `Unit` | Register a cleanup callback |

**RemovalReason** enum passed to `invokeOnRemoval`:
- `NAVIGATION` — entry left the backstack via normal navigation
- `RESET` — entry was cleared by `Store.reset()`

### AD-22 dispatchAndAwait() — suspending dispatch with result

**Type:** Addition

**Grep:** `dispatchAndAwait`
**File glob:** `**/*.kt`

**Example:**
```kotlin
class AuthLogic(val storeAccessor: StoreAccessor) : ModuleLogic() {

    suspend fun login(credentials: Credentials) {
        val result = api.authenticate(credentials)

        // Suspends until the reducer has processed the action and all
        // middleware has completed — state is guaranteed up-to-date after this line
        storeAccessor.dispatchAndAwait(AuthAction.SetAuthenticated(result.userId))

        // Safe to read the updated state immediately
        val state = storeAccessor.selectState<AuthState>().value
        if (state.isAuthenticated) {
            storeAccessor.navigation { navigateTo(HomeScreen) }
        }
    }
}
```

**Notes:** Use `dispatchAndAwait` when subsequent code depends on the state produced
by the dispatched action. Regular `dispatch()` is fire-and-forget — state may not be
updated by the time the next line executes. `dispatchAndAwait` returns a `DispatchResult`
which can be inspected but is usually ignored.

### AD-23 TitleResource and ActionResource — composable screen metadata

**Type:** Addition

**Grep:** `TitleResource\|ActionResource\|titleResource\|actionResource`

```kotlin
typealias TitleResource = @Composable (() -> String)

```

```kotlin
object ProjectScreen : Screen {
```

```kotlin
val navState by selectState<NavigationState>().collectAsState()
```

### AD-24 popEnterTransition and popExitTransition — back-navigation animations

**Type:** Addition

**Grep:** `popEnterTransition\|popExitTransition`
**File glob:** `**/*.kt`

Every `Navigatable` exposes four transition properties. `enter`/`exit` apply when
navigating **forward**; `popEnter`/`popExit` apply when navigating **back**:

| Property | Direction | Default |
|---|---|---|
| `enterTransition` | Arriving (forward) | required |
| `exitTransition` | Leaving (forward) | required |
| `popEnterTransition` | Re-appearing (back) | `NavTransition.None` |
| `popExitTransition` | Leaving (back) | `NavTransition.None` |

**Example — slide right on forward, slide left on back:**
```kotlin
object DetailScreen : Screen {
    override val route = "detail"
    override val enterTransition    = NavTransition.SlideInRight
    override val exitTransition     = NavTransition.SlideOutLeft
    override val popEnterTransition = NavTransition.SlideInLeft   // re-enters from left
    override val popExitTransition  = NavTransition.SlideOutRight // exits to the right
}
```

**Notes:** If `popEnterTransition` / `popExitTransition` are left as `null` the system
falls back to `NavTransition.None` (instant). Set them explicitly whenever the screen
uses directional slide transitions to avoid jarring reverse animations on back.

### AD-26 PendingNavigation — structure and UI access

**Type:** Addition

**Grep:** `pendingNavigation\|PendingNavigation`
**File glob:** `**/*.kt`

When a guard returns `GuardResult.PendAndRedirectTo`, the original intent is stored
in `NavigationState.pendingNavigation` as a `PendingNavigation` object.

**PendingNavigation fields:**

| Field | Type | Description |
|---|---|---|
| `route` | `String` | The original target route that was blocked |
| `params` | `Params` | Navigation params that were passed with the original intent |
| `metadata` | `Map<String, Any>` | Arbitrary key/value pairs set in the guard |
| `displayHint` | `String?` | Human-readable hint to show in the redirect UI |

**Providing metadata and a hint from the guard:**
```kotlin
GuardResult.PendAndRedirectTo(
    navigatable = LoginScreen,
    displayHint = "Sign in to access your workspace",
    metadata = mapOf("source" to "workspace-deep-link")
)
```

**Reading it in a Composable (e.g. the login screen):**
```kotlin
@Composable
fun LoginScreen.Content(params: Params) {
    val navState by selectState<NavigationState>().collectAsState()
    val pending = navState.pendingNavigation

    if (pending != null) {
        Text(pending.displayHint ?: "Sign in to continue")
    }

    // After successful login:
    LaunchedEffect(isLoggedIn) {
        if (isLoggedIn) store.resumePendingNavigation()
    }
}
```

**Notes:** `pendingNavigation` is `null` when there is no stored intent.
`resumePendingNavigation()` clears it and re-navigates to the stored route. If the
user dismisses the login flow without authenticating, call
`store.dispatch(NavigationAction.ClearPendingNavigation)` to discard the intent.

### AD-27 screenRetentionDuration — screen memory lifetime after removal

**Type:** Addition

**Grep:** `screenRetentionDuration`
**File glob:** `**/*.kt`

**Example:**
```kotlin
val navModule = createNavigationModule {
    rootGraph { ... }

    // Keep screens in memory for 30 seconds after leaving the backstack.
```
